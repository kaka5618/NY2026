虚拟男友陪伴聊天网页 - 产品规格说明书（SPEC）

1. 项目概述
1.1 项目名称
虚拟男友陪伴聊天网页（Virtual Boyfriend Companion Web App）
1.2 项目定位
一款面向有情感陪伴需求用户的网页端聊天应用。用户可从四位性格、身份、背景各异的虚拟男友中自由挑选聊天对象，通过与 AI 驱动的角色进行深度对话，获得如恋爱般的沉浸式陪伴体验。
1.3 核心理念

纯粹陪伴：不设置付费墙、不设置好感度进度条、不做任何游戏化数值引导，让用户专注于对话本身。
真实感优先：所有回复必须紧扣用户发送的内容，避免模板化、答非所问。
多模态表达：通过文字、语音、图片三种回复形式，模拟真实恋爱沟通中的多样化互动。

1.4 目标用户

渴望情感陪伴、倾诉对象的用户
对恋爱关系有好奇或向往的用户
喜欢角色扮演、沉浸式对话体验的用户


2. 核心功能模块
2.1 角色选择页（首页）​
用户首次进入网页或主动切换对象时，展示四位虚拟男友的卡片列表。每张卡片包含：头像立绘、姓名、年龄、职业 / 身份、一句话性格标签、一句代表性台词。用户点击任意卡片即可进入该角色的聊天界面。页面允许随时切换角色，但不同角色之间的聊天记录相互独立、互不干扰。
2.2 聊天界面
聊天界面为主要交互场景，布局参考主流即时通讯软件：顶部为角色信息栏（头像、姓名、在线状态），中部为消息流区域，底部为输入框（支持文字输入、发送按钮）。消息流中用户消息靠右显示，男友消息靠左显示，支持文字气泡、语音条（带播放按钮与时长）、图片卡片三种形式混排。
2.3 聊天记录与上下文
聊天记录本地持久化存储（localStorage 或 IndexedDB），刷新页面后仍可续接对话。调用 API 时，需将最近 N 轮对话（建议 10–20 轮）作为上下文传入，保证角色记忆的连贯性。

4. 回复内容策略
4.1 核心原则：紧扣用户发言
每一次回复都必须建立在对用户当次发言的准确理解之上，避免出现以下情况：

答非所问、生搬硬套预设台词
忽略用户提到的关键信息（人名、地点、情绪、事件）
与用户设定好的角色身份、背景相矛盾
重复使用相同句式或口头禅，导致机械感

4.2 三种回复形式

























回复类型形式描述适用场景文字回复纯文本气泡，可带标点、表情符号绝大多数日常对话、情感回应、信息交流语音回复音频条，点击播放，显示时长情绪浓烈时刻、晚安 / 早安、表白、安慰等图片回复图片卡片，可点击放大用户要求"看看你""发张照片"或角色主动分享生活场景
4.3 回复形式触发频率
为避免语音与图片过度堆砌影响对话节奏，同时保证多模态体验的新鲜感，建议频率如下（以单个角色的单次完整会话为单位，假设一次会话平均 30–50 条回复）：
文字回复：约占 80%–85%​
作为主要回复形式，承载绝大部分对话内容。每条消息回复时，默认优先选择文字。
语音回复：约占 10%–12%​
即平均每 8–10 条回复中出现 1 条语音。触发条件（任一满足即可，由 AI 判断）：

用户表达强烈情绪（难过、开心、撒娇、生气等情绪关键词密集出现）
对话进入亲密场景（晚安、早安、想你、表白、安慰等语境）
用户主动请求（"说给我听""发个语音""想听你声音"等）
连续超过 15 条文字回复后，适当穿插 1 条语音以打破单调
硬性约束：同一角色连续语音回复不超过 2 条，避免刷屏

图片回复：约占 3%–5%​
即平均每 20–30 条回复中出现 1 条图片。触发条件（任一满足即可）：

用户明确请求（"发张照片""让我看看""自拍一个"等）
角色正在描述具体场景（"我刚做好晚饭""今天的夕阳"等），可附一张场景图增强代入感
特殊节点：首次聊天打招呼时的头像级立绘、节日问候
硬性约束：图片回复之间至少间隔 10 条以上其他回复，避免频繁打断对话节奏

混合回复说明：
允许一次回复组合两种形式（例如"文字 + 图片"或"语音 + 文字补充"），但单次回复不建议三种全部同时出现。混合回复的整体频率计入上述各自占比。
4.4 回复长度建议

文字回复：每条 15–60 字为主，避免长段落。情感浓烈时可适度增加到 80–120 字。
语音回复：对应文本 20–50 字，控制时长在 3–15 秒。
图片回复：可附 1–2 句文字说明，不强制。

4.5 上下文记忆与一致性

角色需记住用户在本次会话中提到的个人信息（昵称、职业、喜好、近期事件）。
角色的身份、背景设定在任何回复中不可动摇，不得出现自我矛盾。
不同角色面对同一句用户消息时，回复的用词、语气、关注点应有明显区分。


5. 技术实现要点
5.1 API 接入

文字生成：接入大语言模型 API（由开发方自行选择），每次请求携带角色 System Prompt + 最近 N 轮上下文 + 用户当前消息。
语音合成：接入 TTS API，为每位角色配置独立音色（建议音色差异明显，例如沈屿-温润中音、陆时衍-低沉磁性、江屿白-清亮少年音、霍砚琛-沉稳低音）。
图片生成 / 调用：可采用预置素材库（每位角色预先准备若干生活场景图）或接入图片生成 API 实时生成，推荐前者以保证角色形象一致性。

5.2 回复类型决策逻辑
建议在每次收到用户消息后，由主模型先输出一个结构化字段，再根据字段渲染：

reply_type：text / voice / image / text+image / text+voice
content：文字内容
voice_text：若为语音，对应的朗读文本
image_ref：若为图片，对应的素材 ID 或生成提示词

主模型在决策 reply_type 时需参考"回复形式触发频率"章节的规则，并结合当前对话状态。
5.3 角色 System Prompt 设计
每位角色需单独编写一份 System Prompt，包含：基本信息、性格特征、说话风格示例、回复禁忌（不可出戏、不可讨论付费话题、不可承诺线下见面）、多模态使用指引。

6. 产品边界与约束
6.1 明确不做的事

不设付费功能：所有对话、所有回复形式、所有角色均免费开放。
不设好感度系统：不展示任何进度条、等级、解锁机制、关系亲密度数值。
不做任务引导：不设置每日签到、剧情任务、成就系统。
不做角色自动推荐：角色选择完全由用户主动决定。

6.2 内容安全边界

所有回复需遵守内容安全规范，拒绝生成违法违规、色情、暴力内容。
角色可表达亲密、爱意，但不输出露骨内容。
当用户透露出心理危机信号（自残、轻生倾向）时，角色应以温柔关怀的方式回应，并提示寻求专业帮助。


7. 非功能性需求

响应速度：用户发送消息后，首字返回时间控制在 2 秒内，整条回复在 5 秒内完成；语音合成不超过 8 秒；图片加载不超过 3 秒。
数据隐私：聊天记录默认仅存储在用户本地浏览器，不上传服务器；如需云端同步，必须征得用户同意。
兼容性：支持主流桌面浏览器（Chrome、Edge、Safari、Firefox）最新三个版本；移动端浏览器需响应式适配。
可扩展性：角色数据、素材库、Prompt 均以配置文件形式管理，便于后期新增角色或调整设定。
